Typical Process of Establishing and Operating a Subscription with the PUSH Protocol¶

Implementation of the Subscriber System¶

The subscriber implements a web service capable of receiving HTTP POST requests at the subscriber’s URL (specific to a particular subscription). The request body contains the provided message, which the subscriber stores for further processing.

This web service is then exposed at a publicly accessible address. It is desirable for the domain to be secured using TLS (thus accessible via the HTTPS protocol) and to implement authentication through Basic authentication (per RFC 7617). The subscriber should verify, for example, using tools like curl, that the service is accessible and functional. For further purposes, they should make a note of the target URL and the username and password for authentication.

Subscription Establishment¶

The subscriber visits the Subscription Registry, creates an account if necessary, and proceeds to create a new subscription. For the selected source, which offers the desired content and uses the PUSH protocol, they primarily enter the URL to which the provider will send data and provide a username and password. These details correspond to the URL and username and password that the subscriber’s system is ready to serve.

Activation of Delivery¶

Upon activating the established subscription (which may take some time in the case of internal approval by the provider), the provider sends the first batch of messages containing records that are valid at that moment.

Ongoing Delivery of Changes¶

From this point onwards, the provider’s system awaits an internal update of the provided content, and as soon as this occurs, it sends the subscriber a new message.
