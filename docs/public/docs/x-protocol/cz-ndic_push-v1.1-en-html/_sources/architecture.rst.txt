============
Architecture
============

This chapter describes the key systems and actors involved in providing information using the PUSH protocol.

.. uml::
   :caption: Key Systems and Actors
   :align: center

    @startuml
    folder publisher as "Provider" {
        actor pub_admin as "Administrator"
        boundary reg_sub as "Subscription Registry"
        database pub_db as "Database\nwith Records"
        control pub_proc as "Delivery Process"
    }
    folder receiver as "Subscriber" {
        actor recv_admin as "Administrator"
        boundary recv_url as "Subscriber's URL"
        database recv_db as "Database\nwith Records"
    }
    component event_src as "Information Source"
    component recv_app as "Subscriber's Application"
    event_src -> pub_db: Update
    pub_admin -> reg_sub
    recv_admin -> reg_sub
    recv_admin --> recv_url
    pub_proc -> recv_url
    reg_sub --> pub_proc
    pub_db -> pub_proc
    recv_url -> recv_db
    recv_db -> recv_app
    @enduml

External Actors
===============

External actors do influence or utilize the described protocol in some way but are not the main focus of this specification.

An external information source directly or indirectly updates records in the provider's database.

The subscriber's application utilizes the information received from the provider in some manner on the subscriber's side.

Provider
========

The provider ensures the operation of the system and the information provisioning processes.

The provider's administrator approves subscriptions in the subscription registry created by a subscriber.

The subscription registry configures the delivery process.

The database with records stores current information and notifies the delivery process of changes.

The delivery process ensures the delivery of messages with records to subscriber's URLs for approved subscriptions.

Subscriber
==========

The subscriber requests the delivery of information (by creating a subscription) and manages the systems and processes for their reception.

The subscriber's administrator configures the URL (subscriber's URL) in the subscription registry for data reception.

The subscriber's URL (or handling system) receives messages with change records, stores them in the record database, and confirms their reception to the provider.

The database with records (for subscribers) stores received records to provide awareness to its applications of currently valid records.
