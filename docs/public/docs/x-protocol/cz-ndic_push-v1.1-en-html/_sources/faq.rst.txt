=====================
Questions and Answers
=====================

How do I know if the downtime has exceeded the tolerated period?
================================================================

Indicative (non-binding) parameters:

- Acceptable duration of (subscriber's system) unavailability: 24 hours
- Maximal interval for repeated delivery attempts: 15 minutes

If, after the recovery of the subscriber's system, no message is delivered within 15 minutes, the tolerated downtime has likely been exceeded.

The subscriber's system was briefly unavailable, how should I proceed?
======================================================================

After the recovery of the subscriber's system, the subscriber should take the following steps.

Wait for approximately 15 minutes. If at least one message is received within this time, it was a tolerated downtime, and everything should work without further action.

If messages are not coming in, there has been a permanent blockage of the subscription. It is necessary to use the subscription recovery feature in the subscription registry (this is not the creation of a new subscription, just a reactivation).

If all the received information has a relatively short validity period (e.g., 10-20 minutes), the risk of expired messages on the subscriber's side or missing messages is practically zero, and no action is required.

However, if the information has a longer-term validity, a so-called subscription reset is necessary. For more information, see 'What is a subscription reset and how to use it?'_.

How to handle a situation where the subscriber's database contains records that should no longer exist?
========================================================================================================

If there is suspicion that there are records on the subscriber's side that appear to be outdated (their validity should have been reduced, but this did not happen), there may be two reasons:

- The subscriber failed to process some received records correctly (e.g., a program error).
- The subscriber's system was temporarily unavailable with a subsequent suspension of delivery. During the suspension, the validity of some records was reduced, but the subsequent resumption of delivery did not include these changes (naturally).

In this case, a subscription reset is required. For more information, see 'What is a subscription reset and how to use it?'_.

How to address a situation where the subscriber's database is missing records that should be there?
===================================================================================================

If there is suspicion that records are missing on the subscriber's side, the cause is usually an error in processing received records on the subscriber's side.

Note: Resuming a subscription after it has been suspended sends only valid messages. Therefore, the resumption of a suspended subscription should not be the cause of this problem.

In this case, a subscription reset is required. For more information, see 'What is a subscription reset and how to use it?'_.

.. _reset:

What is a subscription reset and how to use it?
===============================================

A subscription reset is an event that terminates the sending of previously sent messages and reinitializes the entire sending process, so messages are sent again as if a new subscription had been created (and approved).

In practice, this means that after the reset, messages with all currently valid records are immediately sent, and subsequently, change messages are sent when there are modifications or new records.

The reset can be initiated for an active subscription in the subscription registry.

It is desirable that the subscriber, as part of the reset, removes from their database records that were received earlier and now appear to be currently valid. This is particularly important if there is a suspicion that long-valid records are persisting in the subscriber's database and should probably have already been terminated.
