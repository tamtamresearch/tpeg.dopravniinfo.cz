============================
Appendix A: Message examples
============================

accident.xml - accident of two vehicles
=======================================

Publication contains exactly one message about an accident of two vehicles, where at least one of them is a truck.

.. literalinclude:: ../../samples/accident.xml
   :language: xml

Root element and publication envelope
-------------------------------------

Root element `d2LogicalModel` declares, that this message is using DATEX II.

Element `exchange` contains provider identifier.

Element `payloadPublication` is an envelope for actual information payload containing general information and one or more elements `situation`, (in our case there is one).

Attribute `xsi:type="SituationPublication"` in element `payloadPublication` determines, that this envelope is of type Situation publication.

Element `feedType` states publikacation schema by its uri.

Element `publicationTime` states publication time of this publication as a whole.

Element `publicationCreator` identifies creator of the publication.

Description of one specific situation
-------------------------------------

Element `situation` describes a particular event, more particular events means more `situation` elements. Each `situation` element may contain one or more individual sub-messages, elements `situationRecord`. In this case occurs in the element `situation` just once.

Element `situationVersionTime` sets the time of the publication of this version of the situation.

Element `headerInformation` describes how to handle the information described.

Event description (what happened)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Attribute `xsi:type="Accident"` of element `situationRecord` defines event
type.  In DATEX II model it can be verified, that given attribute is of
type of descendant of class `SituationRecord` in following descendant sequence `Accident` -> `TrafficElement` -> `SituationRecord`.

Element `probabilityOfOccurrence` states by value `certain`, that the event has certainly happened.

Element `accidentType` defines the type of accident. In this case value
`accidentInvolvingHeavyLorries` states, that a heavy truck is involved in
this accident.

Element `vehicleInvolved` partially repeats previous information by giving
characteristics of the participating vehicle being `lorry`.

Time validity
^^^^^^^^^^^^^

Estimated time validity is given in a structure `validity` by values of validity interval start `overallStartTime` and validity interval end `overallEndTime`.

Event location
^^^^^^^^^^^^^^

Event location is described in element `groupOfLocations` directly by
embedded description. Reference to predefined locations is not used.
Attribute `xsi:type="Linear"` states, that given element is of type linear location, which can be expressed by different location referencing methods.

The position of linear type is represented by three descriptions. First, using  element `alertCLinear` of type `AlertCMethod2Linear` and then described using Global Network links in element `globalNetworkLinear`.

All location referencing methods used are describing the same location. Because the position of event is located within a ALERT-C segment, the `Linear` location type has to be used, with the result that `startPoint` and `endPoint` in the `globalNetworkLinear` element will be the same.

Aditional content
^^^^^^^^^^^^^^^^^

The document states time of its first version in `situationRecordCreationTime` and time of current version in `situationRecordVersionTime` for each `situationRecord`.

Element `source` lists the the creator of this information, who is responsible for its accuracy.

brokendownvehicles.xml - two broken down lorries
================================================

Publication contains information about two broken down lorries,
which fully block traffic on given road.

.. literalinclude:: ../../samples/brokendownvehicles.xml
   :language: xml

Publication envelope, time validity and location
------------------------------------------------

Publication envelope, time validity and location is using identical structures as previous example.

The only difference is in description of one concrete situation in element `situationRecord`.

Event description (what happened)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Attribute `xsi:type="VehicleObstruction"` of element `situationRecord`
defines the type of event. In DATEX II model it can be verified, that given attribute is of type of descendant of class `SituationRecord` in following descendant sequence `VehicleObstruction` -> `Obstruction` -> `TrafficElement` -> `SituationRecord`.

Element `probabilityOfOccurrence` has value `certain` which means, that the event certainly happened.

Element `impact` states in `trafficConstrictionType`, that the accident
resulted in the road being totally impassable.

Element `numberOfObstructions` by value of `2` states that there are 2 obstacles.

Element `vehicleObstructionType` by value `brokenDownVehicle` specifies the type
of obstacle being "broken down vehicles".

Element `obstructingVehicle` extends the previous information by value `lorry`, which means, that lorry or lorries are involved.

Time validity
^^^^^^^^^^^^^

As in previous example.

Event location
^^^^^^^^^^^^^^

It differes from previous example in using one more method, instead of two, thre are used. First, using  element `alertCLinear` of type `AlertCMethod2Linear`, then using a general description of the position on the road in element `linearWithinLinearElement` and then described using Global Network links in element `globalNetworkLinear`.

The same case occurs as in the previous example, with respect to the ALERT-C location that is linear (can not be point), methods are also selected as linear.

All location referencing methods used are describing the same location.

Aditional content
^^^^^^^^^^^^^^^^^

As in previous example.
