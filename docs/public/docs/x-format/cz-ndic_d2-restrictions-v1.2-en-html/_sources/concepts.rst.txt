========
Concepts
========

Described format allows publishing information about road actual and planned closures and restrictions.

Publication type SituationPublication
=====================================

Format is based on DATEX II publication of type `SituationPublication`
[16157-3]_. Profile uses only small subset of possible structures.

Event description
=================

Each event is described by the `situation` element, the described situation can evolve in time, so the element is versioned and has its unique identifier. 

The `situationRecord` element can occur more than once in the `situation` element (but each time of a different type), it is usually information related to one event, situation. The `situationRecord` element can evolve over time, so the element is versioned and has its unique identifier derived from the identifier of the parent situation.

The traffic situation publication in particular includes:

- Road manager actions

  - Road works
  - Road management

    - Road closures
    - Detours
    - Speed limits
    - Change of traffic arrangements
	
.. note::

	This publication does not include any status information.
	
.. important::

	This publication is fully included in the d2-common publication where road closures and road works are published along the other information.
	
Only the structure provided by the `OperatorActions` class is used to describe events. See the W3C XML Schema or [d2ref]_ as appropriate for the exact definition.

The impact of the event is described in the `impact` element, either in the form of expected delay or an estimate of the residual capacity of the road (when closure = 0).

Any additional supplementary information on the event is given by an html link in the `urlLink` element.

The indication that this is a safety-related event according to the Delegated Regulation 886/2013 is given in the `safetyRelatedMessage` tag.

Location description
====================

Each `SituationRecord` has its own position description, which may (or may not) differ within a `Situation`. For example, one `SituationRecord` describing a point source problem that resulted in a linear problem (the second `SituationRecord`).

A location tends to be described in a `SituationRecord` by multiple methods, all referring to the same slice of reality.

=============
Time validity
=============

Time validity is stated independently for every single event (`SituationRecord`) described in a situation and is defined by a time interval between two time instants. The class Validity uses exclusively an instance of class `OverallPeriod` in the role of `validityTimeSpecification` and elements `overallStartTime` and `overallStopTime` to define start and end of the interval.

Setting a week-day and recurring time validity within the specified period, e.g. for recurring intermittent nightly tunnel closure, is not allowed. For such information, splitting to more messages valid in respective times is used.

Used time validity intervals may fall into past, present or future.

====================
Location description
====================

When describing locations, nested descriptions of instances of the `Location` class are **always** used, namely the descendants of the `NetworkLocation` class. 

The `Linear` class is always used to describe a position, regardless of the fact that it may be a point position, e.g. an accident, expressible by a single coordinate. The reason for this is to unify descriptions between multiple methods. Area type position descriptions and separate point position descriptions are not used.

.. literalinclude:: ../../samples/restriction.xml
   :language: xml
   :linenos:
   :lineno-match:
   :start-after: </urlLink>
   :end-before: <alertCLinear

Multiple methods can be used to describe a location, depending on their availability in a given location. At least one description must be given.  

The following description lists the methods used for location description. For more details, see the `en-ndic_d2-predefined-location-set-v1.1` profile documentation. 

Location description by means of ALERT-C
========================================

The method describes the position using ALERT-C codes, class `alertCLinear` of type `AlertCMethod2Linear` is used. The use of this method depends on the presence of an ALERT-C location at a given location.

The `alertCLinear` method uses a primary location (`alertCMethod2PrimaryPointLocation` element), a secondary location (`alertCMethod2SecondaryPointLocation` element) to describe the event. Of course, the method also contains elements allowing to identify a specific location table.

The primary location is, according to the ALERT-C principle, the place where the accident occurred or where the driver leaves the problem area. The secondary location is the end of the convoy or where the driver first enters the problem area. 

In order to find out the direction of the situation, the `alertCDirectionCoded` element is used, which takes the value `positive` if the direction of the traffic flow agrees with the coding of the given route. 

.. important::

	WARNING, the meaning of this attribute is therefore exactly the opposite of the `direction` attribute in RDS-TMC, which is used to find the secondary location.


The `alertCLinear` element may occur multiple times in the situation description for long events that must be described by more than one ALERT-C path. This is because ALERT-C only allows locations to be chained within a single path.

..  note::

	The `alertCLinear` method is also used to describe a **point location** (even the inside of an intersection or tunnel, i.e. a short segment). When primary and secondary locations are the same. The direction can be read from the `alertCDirectionCoded` element).

Location description using mileage
==================================

The method uses `linearWithinLinearElement` for the description of the position using mileage. The `roadNumber` element describes the number of the affected road, the `directionRelativeOnLinearSection` element describes the direction of the affected road, and the `fromPoint` and `toPoint` elements describe the distance of the start and end of the event from the start of the road. 

.. note::

	The `linearWithinLinearElement` method is also used to describe a **point location** (even the inside of an intersection or tunnel, i.e. a short section). When the offsets `fromPoint` and `toPoint` refer to a segment with zero length.

Location description using the Global Network extension
=======================================================

Native method of position input in NDIC using **Extension of DATEX II model type B**.

The method describes, in the `globalNetworkLinear` element, the location using the segments `linearWithinLinearGNElement` of the Global Network (GN) road network, identified by the `networkVersion` version. Each segment contains a sequence, an identifier referring to the GN, and a relative truncation (for the first and last segment). Furthermore, the position is bounded by the elements `startPoint` and `endPoint` in coordinates (here S-JTSK).

.. note::

	The `globalNetworkLinear` method is also used to describe a **point location** (even the inside of an intersection or tunnel, i.e. a short segment). The `startPoint` and `endPoint` positions contain the same coordinate.

   :language: xml
   :linenos:

Location description using OpenLR extension
===========================================

The OpenLR method in the `openlrExtendedLinear` element, uses a routing algorithm to describe the location and thus allows on-the-fly localization.

The method uses the reference points `openlrLocationReferencePoint` and `openlrLastLocationReferencePoint` containing coordinates, road direction, and a road type. These reference points are used as the start and end to find the road which is further refined using offsets to the actual extent of the event. In case 2 points are not enough the method allows to use intermediate points. 

.. note::

	To describe a **point location** (even the inside of an intersection or tunnel, i.e. a short segment) the `openlrExtendedLinear` method is also used, when offsets refer to a segment with zero length.
