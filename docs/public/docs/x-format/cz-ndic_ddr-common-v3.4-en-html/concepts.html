Concepts¶

This section highlights main concepts, used by this format. For complete description refer to schemas with definitions in comments.

In this section are provided some snippets of real XML documents. For more examples, see Appendix A: Message examples

XML Document Sections¶

The XML Document is built using following elements and sections:

  • DOC (document root)

  • INF (metadata)

  • MJD (container for messages)

  • MSG (message)

Document root (DOC)¶

The DOC element represents single publication (one instance of information message).

The xmlns attribute refers to the version of the schema used.

The id attribute specifies the identity of the publication as a whole.

The country attribute specifies the code of the country in which the events contained in the message are located. There must not be events from more than one country in one report.

1<?xml version="1.0" encoding="UTF-8"?>
2<DOC
3    xmlns="x-format:cz-ndic_ddr-common-v3.4"
4    id="1f1502dc-86b3-405b-bfb9-dfc6a582d315"
5    country="CZ">

Metadata (INF)¶

INF element provides meta-information about whole publication:

  • a reference to the ALERT-C event code list (versions) used in the event description

  • reference to the ALERT-C location table (country code, table number and version) used to describe locations

  • reference to the GDF 4.0 map set used (type of network used, network version and country for which the network was created)

  • reference to the code list of organizational units of the Czech Republic UIR-ADR (structure version, data version and issue date)

1  <INF>
2    <DAT>
3      <EVTT version="2.01"/>
4      <LOCT version="8.0" number="25" country="2"/>
5      <SNET type="SN" version="13.12" country="CZ"/>
6      <UIRADR structure="4.2" version="907"/>
7    </DAT>
8  </INF>
9  <MJD>

Message container (MJD)¶

Single traffic messages are located in a parent element MJD. It can contain one or more messages.

Traffic Message (MSG)¶

Particular message is represented by a MSG element. The message itself has following sections:

  • validity in time MTIME

  • textual description of the event MTXT

  • event description MEVT (what has happened)

  • location reference MLOC (where) to linear location

  • description of the position according to the UIR-ADR MDST code list

  • detour description DIVLOC

Message identity and version¶

The message itself has unique identifier id, version, type, provider and message veracity information.

15      <MSG
16          id="eca17d6a-5eea-48e6-b61f-f6060f6ada54"
17          version="1"
18          type="TI"
19          provider="SSU"
20          authorized="true">

The id attribute indicates the identity of the message, it does not change during its updates.

The version attribute indicates the version of the message. Versions start with the number 1 and with each update the version is increased by one.

The type attribute specifies the type of message and can take the values TI for common traffic information and TL for traffic status information (e.g. travel times).

The provider attribute specifies the abbreviation of the provider (originator) of the message.

The authorized attribute indicates whether the event was verified on the spot by the Police. Most messages are verified, the attribute has the value true, but for example, for an unsecured accident location, the value of the attribute is false until on-site verification.

Message time validity (MTIME)¶

The validity of a message over time is specified by a simple validity interval from - to, using time zones.

21      <MTIME>
22        <TGEN>2007-09-26T08:27:19+02:00</TGEN>
23        <TSTA>2007-09-26T08:27:19+02:00</TSTA>
24        <TSTO>2007-10-26T08:27:19+02:00</TSTO>
25      </MTIME>

Message text (MTXT)¶

DDR uses a text description of the event as specified by the operator in the MTXT tag. This text contains both a description of the event itself and its location.

26      <MTXT>Z ulice Vídeňská - Merhautova, do ulice Provazníkova, neprůjezdné, překážka na vozovce, dopravní kolaps v úseku 1 km, mimořádná událost, očekávejte zdržení, po zbytek dne, udržujte vzdálenost mezi vozidly, sledujte zvláštní ukazatele pro objížďku, volný text</MTXT>

Event description (MEVT)¶

DDR uses the ALERT-C event code list to describe the event and respects all the rules given by ALERT-C (multiple event codes, quantifiers).

The MEVT tag contains the message text composed of the values of individual events from the ALERT-C event catalog.

In addition to the codes from the ALERT-C code list, a free text description as entered by the operator is used.

27      <MEVT>
28        <TMCE urgencyvalue="U" directionalityvalue="1" timescalevalue="D"  diversion="true">
29          <EVI eventcode="980" updateclass="5" eventorder="1">
30            <TXUCL>Dopravní uzavírky a omezení</TXUCL>
31            <TXEVC>neprůjezdné, překážka na vozovce</TXEVC>
32          </EVI>
33          <EVI eventcode="102" updateclass="1" eventorder="2">
34            <TXUCL>Dopravní situace</TXUCL>
35            <TXEVC>dopravní kolaps v úseku 1 km</TXEVC>
36          </EVI>
37          <EVI eventcode="1685" updateclass="38" eventorder="3">
38            <TXUCL>Předpověď zdržení</TXUCL>
39            <TXEVC>mimořádná událost, očekávejte zdržení</TXEVC>
40          </EVI>
41          <SPI supinfocode="13" supinfotext="udržujte vzdálenost mezi vozidly"/>
42          <DIV diversioncode="61" diversiontext="sledujte zvláštní ukazatele pro objížďku"/>

Location description (MLOC)¶

DDR uses either a point location reference (e.g., an accident), a line location reference (e.g., a column) on a road network, or an area location reference to describe a location.

The following location description methods may be used in DDR (at least one must be used):

  • free text description (as entered by the operator) in the TXPL tag.

  • ALERT-C location codes (primary location, direction and range) in the TMCL tag.

  • GlobalNetwork road network segments reference in the SNTL tag.

  • Mileage with a reference to a road (using its number) in the CHAIN tag

  • Coordinates (WGS-84 and SJTSK) of the beginning (from the driver’s point of view = end of the queue) and the end of the event in the SNTL/COORD tag and. SNTL/COORDEND tag.

  • OpenLR dynamic localization allowing location description using routing points on a road network in the OLR tag.

Note

The position can also be described by several unrelated sections of the road network. This is indicated by the attribute geometry_type with the value non-continuos. Then the TMCL, SNTL, CHAIN, OLR tags may appear more than once. The TMCL tag can appear more than once, even if the location is of the continuos type, in the case of consecutive TMC sections.

47      <MLOC geometry_type="continuous">
48        <TXPL>Z ulice Vídeňská - Merhautova, do ulice Provazníkova</TXPL>
49        <TMCL primarycode="16366" extent="1" direction="+" roadid="332"/>

Location referencing¶

Following is a selection of most used location referencing methods.

Location description by means of ALERT-C¶

The first method describes position by ALERT-C codes, by element TMCL. The use of this method depends on the presence of the ALERT-C location at a given location.

The TMCL method uses the primary location (attribute primarycode), extent (attribute extent) and direction (attribute direction) element to describe the event. Used location table is described at the level of document in element INF, so all messages MSG in the document must use same location table.

The primary location is, by the ALERT-C principle, the place where the accident occurred or where the driver leaves the problem area. The secondary location is the end of the traffic queue or the place where the driver first enters the problem area.

49        <TMCL primarycode="16366" extent="1" direction="+" roadid="332"/>

Location description using the Global Network¶

The second method uses for location description identifiers of affected sections of the road network, specified by Global Network (GN) database, and event start and (end) coordinates (in the S-JTSK and WGS-84 reference system).

The COORD element describes the coordinates of the “start” of an event in the direction of traffic. In case of a linear event, the coordinates indicate the start of the event in terms of the direction of traffic, the place to which the driver arrives first, in TMC it is the secondary location.

The COORDEND element describes the “end” coordinates of the event in the direction of traffic. In case of a linear event, the coordinates indicate the end of the event in terms of the direction of traffic, i.e. the place where the driver leaves the problem area, in TMC it is the primary location

The STEL element describes specific sections of the GN network by their identifier el_code, directionality of the section el_dir and by the order of their appearance order, as they follow each other on the network.

This is native means of location referencing by NDIC.

50        <SNTL count="5">
51            <COORD x="-599220" y="-1163113" lat="49.1729875" lon="16.5970553" />
52            <COORDEND x="-599240" y="-1163160" lat="49.1725478" lon="16.5968519"/>
53          <STEL el_code="725704" el_dir="+" order="1"/>
54          <STEL el_code="725706" el_dir="-" order="2"/>
55          <STEL el_code="638420" el_dir="+" order="3"/>
56          <STEL el_code="638412" el_dir="-" order="4"/>
57          <STEL el_code="638377" el_dir="+" order="5"/>
58          <STEP begin="0.545454561710358" end="0.545454531908035"/>
59        </SNTL>

Location description using OpenLR¶

The fourth method uses the OpenLR routing method for dynamic location description [http://www.openlr.org/method.html].

This makes it possible to mark significant points of the route on the provider’s map on which the traffic problem is located and, together with selected attributes of these points and related road attributes, to transmit them to the recipient. According to the information provided, the end user device then tries to route the location on its map; if succeeds, it can find the exact location of the problem on this route.

The OLR element encapsulates the information needed by the OpenLR method.

61        <OLR>
62          <XMLLocationReference xmlns="http://www.openlr.org/openlr">
63            <LineLocationReference>
64              <LocationReferencePoint>
65                <Coordinates>
66                  <Longitude>16.61287339388525</Longitude>
67                  <Latitude>49.20643828368735</Latitude>
68                </Coordinates>
69                <LineAttributes>
70                  <FRC>FRC7</FRC>
71                  <FOW>SINGLE_CARRIAGEWAY</FOW>
72                  <BEAR>306</BEAR>
73                </LineAttributes>
74                <PathAttributes>
75                  <LFRCNP>FRC7</LFRCNP>
76                  <DNP>854</DNP>
77                </PathAttributes>
78              </LocationReferencePoint>
79              <LastLocationReferencePoint>
80                <Coordinates>
81                  <Longitude>16.60674140075723</Longitude>
82                  <Latitude>49.212793571232076</Latitude>
83                </Coordinates>
