========
Concepts
========

Described format allows publishing of predefined locations used for referencing of status information (travel times, level of service, tunnel status, parking occupancy, weather station reports etc.).

Publication type PredefinedLocationPublication
==============================================

Format is based on DATEX II publication of type `PredefinedLocationPublication`
[16157-3]_. The profile uses most of data structures, provided by this type of publication and extends it by proprietary location referencing.

  .. literalinclude:: ../../samples/predefined_location_set.xml
   :language: xml
   :linenos:
   :lineno-match:
   :end-before: <predefinedLocationContainer


Predefined location description (point or linear segment)
=========================================================

*Location* is a point or linear segment on road network. Location
type area is not used. The locations are described by means of several methods, each `predefinedLocationContainer` describes one location (by one or more methods). Element `predefinedLocationContainer` is unanimously identified by unique identifier and version. 

Each `predefinedLocationContainer` could contain a description in an element `PrefefinedLocationName`.

  .. literalinclude:: ../../samples/predefined_location_set.xml
   :language: xml
   :linenos:
   :lineno-match:
   :start-after: </predefinedLocationContainer>
   :end-before: </payloadPublication>

=============
Time validity
=============

The publication start of validity is defined by `publicationTime` for all predefined locations in the publication. No other time record is contained in the publication. In case of new publication due to changes to predefined location set, old location remains the same, new locations get new id and updated locations got its version increased.

====================
Location referencing
====================

All locations in this publication are predefined and each is contained within element `PrefefinedLocationContainer` globally identified by its identifier and version (these are used to reference the location from other publications). Element `location` (descendant of `NetworkLocation`) with type `Point` or `Linear` contains description of one physical location in one or more location referencing methods.

Three location description methods for a linear location and four location description methods for point location can be used to describe the location. The method use depends on method availability at the location. At least one location description should be used.

Location description by means of ALERT-C
========================================

The method describes position by ALERT-C codes, by class `alertCLinear` of the `AlertCMethod2Linear` type for linear locations and by class `alertCPoint` of the `AlertCMethod2Point` type for point locations. Other methods of ALERT-C description are not used. The use of this method depends on the presence of the ALERT-C location at a given location.

The `alertCLinear` method uses the primary location (element `alertCMethod2PrimaryPointLocation`), secondary location (element `alertCMethod2SecondaryPointLocation`) and direction (`alertCDirection`) element to describe the event. Of course, the method also includes elements that allows to identify a particular location table.

  .. literalinclude:: ../../samples/predefined_location_samples_set.xml
   :language: xml
   :linenos:
   :lineno-match:
   :start-after: <!-- Sample of alertCLinear begin -->
   :end-before: <!-- Sample of alertCLinear end -->

Element `alertCLinear` can appear more than once in a situation location description. This could happen for long events that must be described by more than one ALERT-C route. ALERT-C allows to chain (reference between) locations only within one route.

The primary location is, by the ALERT-C principle, the place where the accident occurred or where the driver leaves the problem area. The secondary location is the end of the traffic queue or the place where the driver first enters the problem area.

This method is not optimal because the RDS-TMC uses to describe the event location the primary location, direction and extent, looking up the secondary and intermittent locations in the location table. Therefore, if the recipient systems are adapted to RDS-TMC event reconstruction, it will be necessary to modify them.

The `alertCPoint` method is used to describe the point location (the inside of a junction or a tunnel, a short section). This method uses the primary location (element `alertCMethod2PrimaryPointLocation`) and the direction (`alertCDirection`) to describe the event. Of course, the method also includes elements that allow to identify a particular location table.

  .. literalinclude:: ../../samples/predefined_location_samples_set.xml
   :language: xml
   :linenos:
   :lineno-match:
   :start-after: <!-- Sample of alertCPoint begin -->
   :end-before: <!-- Sample of alertCPoint end -->

However, if it is necessary to express the short location covered by one ALERT-C point also by other methods, then the `alertCLinear` method has to be used (where the secondary location is equal to the primary), since the methods describing one location must be of same type e.g. point or linear.

Location description using a textual description of the road network
====================================================================

The method uses the "address" of the street or road (name, area, number, etc.) and the distance of the event location from the beginning and the end of the road. For description of linear locations, the `linearWithinLinearElement` class is used and for point locations the `pointAlongLinearElement` class is used. The use of this method depends on the presence of a proper description of the road (ideal for motorways).

The `roadNumber` element describes the number of the affected road, element `DirectionRelativeOnLinearSection` describes the direction of affected traffic, and the `fromPoint` and `toPoint` elements describes distance of the start and the end of the event from the beginning of the road.

Location description using the Global Network extension
=======================================================

This is a DATEX II model extension type B.

The method uses for location description identifiers of affected sections of the road network, specified by Global Network (GN) database, and event start and end coordinates (in the S-JTSK reference system). The last and initial segments are truncated by the relative distance of the start and end points of the event relative to the GN segment.

The `globalNetworkLinear` class is used to describe the linear location and `pointWithinLinearGNElement` for point location description.

The `globalNetworkLinear` element describes the location of one event on the network. It contains the network version identifier `networkVersion`, the points bounding the beginning and end of the event (`startPoint` and `endPoint` in coordinates, here S-JTSK). And one or more `linearWithinLinearGNElement` elements that describe specific segments of the GN database by their direction, identifier, location order, and relative truncation (for the first and last sections).

This is native means of location referencing by NDIC.

Location description using Coordinates
======================================

The method uses geographic coordinates to describe a situation location. Two methods may be used based on used coordinate reference system. 

Method `pointByCoordinates` is used for description of position in WGS-84 coordinate system.

Method `pointByMultiCoordinates` is used to describe location by one ore more coordinate systems, one of which being Czech system S-JTSK and second WGS-84.
This is a DATEX II model extension type B.
